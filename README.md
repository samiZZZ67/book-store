# Simple PDF Reading Site

Small Django project for uploading PDFs, requesting per-book access, and reading approved PDFs inside the website.

The default local database is SQLite, so it runs without installing a database server. Production deployments use PostgreSQL through `DATABASE_URL` or Azure PostgreSQL Service Connector variables.

Admins can add a JPG, PNG, GIF, or WebP thumbnail when uploading a PDF, and can update or remove each book thumbnail later from the admin dashboard.

## Setup

1. Install dependencies:

```powershell
python -m venv .venv
.\.venv\Scripts\Activate.ps1
pip install -r requirements.txt
```

2. Optional environment variables:

```powershell
$env:DJANGO_SECRET_KEY="change-this-secret"
```

3. Create the database tables:

```powershell
python manage.py migrate
```

4. Create an admin user:

```powershell
python manage.py create_admin --username admin --email admin@example.com --password admin123
```

5. Run the site:

```powershell
python manage.py runserver
```

Open `http://127.0.0.1:8000/`.

Admin dashboard URL: `http://127.0.0.1:8000/site-admin/`

## Telegram notifications

1. In Telegram, create a bot with BotFather and copy the bot token.
2. Set these environment variables:

```powershell
$env:TELEGRAM_BOT_TOKEN="your-bot-token"
$env:TELEGRAM_WEBHOOK_SECRET="make-a-long-random-secret"
$env:SITE_URL="https://your-public-site.example.com"
```

3. Log in as site admin and open `http://127.0.0.1:8000/site-admin/`.
4. In Telegram Notifications, add up to two admin usernames without `@`. The site will try to set the Telegram webhook automatically.
5. If needed, click **Set Telegram Webhook** in the admin dashboard.
6. Each Telegram admin must send `/start` to the bot from the Telegram account with the same username.
7. If `/start` was sent before the webhook was ready, click **Sync /start Messages** in the dashboard or run:

```powershell
python manage.py sync_telegram_admins
```

8. To inspect the bot, webhook, and chat IDs:

```powershell
python manage.py telegram_status
```

9. On Render, you can also set the webhook from Render Shell after the service is live:

```powershell
python manage.py set_telegram_webhook --base-url https://your-service.onrender.com
```

After that, when a user requests access to a single uploaded book, registered Telegram admins receive a message with the user, email, book title, and dashboard link.

## Reader features

The in-site reader includes these basic features:

- First, previous, next, and last page navigation.
- Direct page number jump.
- Page list sidebar.
- Zoom in, zoom out, and preset zoom levels.
- Fit width and fit page.
- Rotate left and rotate right.
- Fullscreen mode.
- Text search with previous and next result controls.
- Bookmark current page.
- Resume from bookmark or last read page.
- Reading progress bar and percentage.
- Reading timer.
- Light, dark, sepia, and high-contrast themes.
- Keyboard shortcuts for arrows, Home, End, zoom, fullscreen, and search.
- Basic copy/save/print/right-click blocking in the reader page.

## Notes

- PDFs are stored in `private_media/`, not in public static folders.
- Book thumbnails are also stored under private media and served through the Django thumbnail route.
- If `CLOUDINARY_URL` is set, uploaded PDFs and thumbnails are stored in Cloudinary instead of `private_media/`.
- PDF file responses require login, approved access to that specific book, and a short-lived session token generated by the viewer page.
- The site uses PDF.js from a CDN for the custom reader. If your users need offline access, vendor PDF.js locally.
- Browser restrictions cannot fully prevent screen capture, developer-tool access, or all save/print behavior. True DRM requires a dedicated DRM system.

## Cloudinary media storage

Cloudinary stores uploaded files, not the Django database rows. Keep using SQLite locally or PostgreSQL in production for users, access requests, and book records.

To store uploaded PDFs and thumbnails in Cloudinary, set either:

```text
CLOUDINARY_URL=cloudinary://API_KEY:API_SECRET@CLOUD_NAME
```

Or set all three individual values:

```text
CLOUDINARY_CLOUD_NAME=your-cloud-name
CLOUDINARY_API_KEY=your-api-key
CLOUDINARY_API_SECRET=your-api-secret
```

When `CLOUDINARY_URL` is present and `USE_CLOUDINARY_STORAGE` is blank or unset, the app enables Cloudinary automatically. You can also force it with:

```text
USE_CLOUDINARY_STORAGE=1
CLOUDINARY_STORAGE_PREFIX=pdf-library
```

Uploaded PDFs are still served through the protected Django route, so users need login, approval, and a current viewer token to read from the site.

PDF uploads have no Django-side size limit by default. Keep `MAX_UPLOAD_SIZE` blank or set it to `0`, `none`, or `unlimited` for Cloudinary-backed uploads. Cloudinary, Azure App Service, browser timeouts, and temporary disk space can still impose their own practical limits.

Cloudinary can block public PDF delivery depending on the account security settings. The app first tries normal raw delivery and then falls back to a signed Cloudinary download URL. For best PDF support, enable Cloudinary Console > Settings > Security > **Allow delivery of PDF and ZIP files**.

To verify upload, large upload, delete, and credentials from Render Shell:

```bash
python manage.py cloudinary_status --write-test --large-write-test
```

If you already have files in `private_media/`, copy them to Cloudinary under the same names before relying on Cloudinary in production:

```bash
python manage.py migrate_media_to_cloudinary
```

Use `--dry-run` to preview and `--force` to replace existing Cloudinary assets.

## Deploy on Azure App Service

Use a Linux Python App Service. Set the Startup Command to:

```text
bash start.sh
```

Set these App Service environment variables:

```text
DJANGO_ENV=production
DJANGO_SECRET_KEY=<long-random-secret>
SITE_URL=https://<your-app>.azurewebsites.net
MAX_UPLOAD_SIZE=0
```

For the database, either set:

```text
DATABASE_URL=postgresql://USER:PASSWORD@HOST:5432/DBNAME
DB_SSL_REQUIRE=1
```

Or connect Azure Database for PostgreSQL with Service Connector using the Django/password client type, which provides:

```text
AZURE_POSTGRESQL_HOST=<server>.postgres.database.azure.com
AZURE_POSTGRESQL_NAME=<database>
AZURE_POSTGRESQL_USER=<user>
AZURE_POSTGRESQL_PASSWORD=<password>
AZURE_POSTGRESQL_PORT=5432
AZURE_POSTGRESQL_SSL=true
```

`WEBSITE_HOSTNAME` is added by Azure automatically, so the default `*.azurewebsites.net` host is allowed without hard-coding it. For custom domains, set:

```text
DJANGO_ALLOWED_HOSTS=www.example.com,example.com
CSRF_TRUSTED_ORIGINS=https://www.example.com,https://example.com
```

The startup script runs `collectstatic`, migrations, optional admin creation, and Gunicorn. For uploaded PDFs and thumbnails, prefer `CLOUDINARY_URL`; without Cloudinary, Azure deployments default `MEDIA_ROOT` to `/home/site/media`.

## Deploy on Render

This repo includes `render.yaml` and `build.sh`.

1. Push the project to GitHub/GitLab/Bitbucket.
2. In Render, create a new Blueprint Instance from the repository. Render will create the web service and PostgreSQL database from `render.yaml`.
3. Set environment variables on the web service:

```text
TELEGRAM_BOT_TOKEN=your-bot-token
TELEGRAM_ADMIN_USERNAMES=adminusername1,adminusername2
SITE_URL=https://your-service.onrender.com
CLOUDINARY_URL=cloudinary://API_KEY:API_SECRET@CLOUD_NAME
```

4. If `CLOUDINARY_URL` is set, uploads are durable in Cloudinary and the Render disk is only a fallback for non-Cloudinary deployments. Without Cloudinary, keep `MEDIA_ROOT=/var/data/private_media` and the included persistent disk if you want uploaded PDFs to survive restarts and deploys. Render persistent disks require a paid web service; without Cloudinary or a disk, uploaded files are not durable.
5. To create an admin without Render Shell, add these environment variables to the web service and redeploy:

```text
DJANGO_ADMIN_USERNAME=admin
DJANGO_ADMIN_EMAIL=admin@example.com
DJANGO_ADMIN_PASSWORD=change-this-password
```

The startup script creates or updates that admin after migrations run. After the admin is created, you may remove `DJANGO_ADMIN_PASSWORD` from Render and redeploy if you do not want the password stored there.

6. If you have Render Shell, you can also create an admin manually:

```bash
python manage.py create_admin --username admin --email admin@example.com --password change-this-password
```

7. Register the Telegram webhook from the Render Shell:

```bash
python manage.py set_telegram_webhook --base-url https://your-service.onrender.com
```

8. Ask each configured Telegram admin to send `/start` to the bot. The dashboard will show their chat IDs after registration.

### Manual Render service settings

If you created the Render Web Service manually instead of using the Blueprint, set these exactly:

```text
Build Command: bash build.sh
Start Command: bash start.sh
```

If Render shows `Running 'gunicorn pdfsite.wsgi'` in the logs, the Start Command is still wrong.

The start script runs:

```bash
python manage.py collectstatic --noinput
python manage.py migrate --noinput
python manage.py create_admin ... # only when DJANGO_ADMIN_* env vars are set
gunicorn pdfsite.wsgi:application --bind 0.0.0.0:${PORT:-8000}
```

Also create a Render PostgreSQL database and add this environment variable to the web service:

```text
DATABASE_URL=<your Render Postgres Internal Database URL>
```

Use the Internal Database URL, not the External Database URL, when the Django app is also hosted on Render. If `DATABASE_URL` is missing in production, the app now fails fast with a clear configuration error instead of silently using SQLite.

## Optional PostgreSQL

For PostgreSQL, install a PostgreSQL server and the Python driver:

```powershell
pip install "psycopg[binary]"
```

Then set:

```powershell
$env:DB_ENGINE="postgresql"
$env:POSTGRES_DB="simple_pdf_site"
$env:POSTGRES_USER="postgres"
$env:POSTGRES_PASSWORD="your-password"
$env:POSTGRES_HOST="localhost"
$env:POSTGRES_PORT="5432"
```

Run:

```powershell
python manage.py migrate
```
